## 1. 确定 Review 范围

按以下优先级自动识别要 review 的代码：

1. 用户指定了 PR → `gh pr diff <number>`
2. 用户指定了 commit → `git diff <commit>~1 <commit>`
3. 用户说"上一个 commit" → `git diff HEAD~1`
4. 有 staged changes → `git diff --cached`
5. 有 unstaged changes → `git diff`
6. 用户选中了代码（IDE selection）→ 直接使用选中内容

拿到 diff 后，**必须用 ace-tool 检索变更文件的调用链和上下游依赖**，再开始检查。

## 2. 检查维度

逐项检查，跳过没有问题的维度：

- **正确性**：逻辑是否在边界情况和并发场景下仍然正确
- **安全性**：是否存在注入、越权、敏感数据泄露等风险（参考 OWASP Top 10）；作用域是否只开放到够用为止
- **错误处理**：异常路径是否有合理的处理或降级；是否有空 catch；是否有静默失败
- **向后兼容**：是否破坏了现有的接口契约或外部依赖
- **性能**：是否引入了明显的性能退化（如 N+1、无限重渲染、数据膨胀）
- **类型安全**：是否使用了 `any`；类型设计是否合理，是否用 escape hatch 掩盖了真实类型问题
- **命名质量**：命名是否表达意图，是否需要注释才能理解
- **职责边界**：函数是否超过 40 行或承担多个职责（名称含 "and"/"also" 视为信号）；文件是否超过 300 行
- **耦合控制**：是否存在时序耦合、全局状态耦合、数据结构耦合、循环依赖
- **复用与抽象**：是否有重复逻辑（出现 3 次须抽取）；是否存在为未来需求创造的过度抽象

## 3. 输出格式

先输出总览摘要，再按严重性从高到低列出问题。

### 总览

格式：`发现 X 个必须修复，Y 个建议修复，Z 个可选优化`（数量为 0 的级别省略）

### 严重性标注

- `[必须修复]` — 阻断级问题：会导致 bug、安全漏洞、数据丢失
- `[建议修复]` — 有明确风险或隐患，不修会埋坑
- `[可选优化]` — 代码质量提升，非紧急

### 每条格式

```
[严重性] 维度 | 简述问题
文件:行号 — 具体说明
```

### 规则

- 只输出有问题的条目，没问题的维度不列
- 每条必须引用具体文件和行号
- 不要输出格式化、注释补充、import 排序等 linter/formatter 能处理的问题
- 如果所有维度都没问题，输出「无问题」即可
