## 合理化防御

在开始调试之前，审视自己是否在找借口跳过流程：

| 你想说的 | 为什么是错的 |
|---|---|
| "我大概知道问题在哪，直接改" | 大概 ≠ 确定，盲改浪费时间且掩盖真正原因 |
| "先试一下，不行再查" | 试了不行你会再试，三轮下来已经偏离根因 |
| "这个错误很常见，肯定是 X" | 常见原因不代表是这次的���因，先验证 |
| "改动很小，不需要这么正式" | 小改动引发的 bug 往往更难排查，正因为小才要严谨 |

## 阶段一：根因调查

在尝试任何修复之前，必须完成以下步骤：

1. **完整阅读错误信息** — 不要只看第一行，完整读 stack trace
2. **复现问题** — 确认能稳定复现；不能复现的 bug 不要猜改
3. **检查最近变更** — `git log` / `git diff` 查看相关文件的近期改动
4. **追踪调用链** — 用 ace-tool 检索出错函数的调用方和依赖，理解数据流向
5. **缩小范围** — 在多组件系统中，逐层加诊断日志定位问题出在哪一层

**产出**：明确的根因假设，不是"我觉得可能是 X"，而是"日志/证据显示问题出在 X，因为 Y"。

## 阶段二：模式分析

1. 在项目中找到**工作正常的类似代码**（同类功能、同类调用）
2. 逐一对比出问题的代码与正常代码的差异
3. 差异点就是修复线索

## 阶段三：假设验证

- 科学方法：**一次只改一个变量**
- 每次修改后立即验证（跑测试、看日志、确认效果）
- 验证失败则回退修改，尝试下一个假设
- **禁止同时改多处然后一起验证** — 这样你不知道哪个改动生效了

## 阶段四：最小修复

1. 只修复根因，不顺手重构周边代码
2. 修复后展示验证证据（测试通过、错误消失、功能正常）
3. 确认修复没有引入新问题（跑完整测试套件或检查相关功能）

## 止损规则

**连续 3 次修复尝试失败后，必须停止。**

此时不要再试第 4 次，而是：
1. 回退所有修改到初始状态
2. 重新审视问题 — 是否在错误的层面上找原因？
3. 质疑假设 — 你以为的根因可能不是真正的根因
4. 向用户报告当前发现，讨论后再继续
